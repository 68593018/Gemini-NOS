# NOS Scheduler Performance Tuning & Hybrid Busy-Polling

## 1. 架构演进与设计目标

在追求千万级甚至百万级 TPS (Transactions Per Second) 和纳秒级时延的高性能通信场景中，传统的基于互斥锁 (Mutex) 和内核阻塞队列 (如 pipe, eventfd + epoll_wait) 的调度架构面临严重的性能天花板：

*   **系统调用开销**：`write` 和 `read` 频繁触发用户态到内核态的切换。
*   **上下文切换开销**：内核调度器挂起和唤醒线程导致 L1/L2 Cache 失效。

为突破此瓶颈，NOS 调度器引入了**混合轮询 (Hybrid Busy-Polling)** 架构，结合了 DPDK 等高性能系统的设计理念，实现“零系统调用 (Zero-Syscall)”的高频数据通路。

## 2. 核心优化机制

### 2.1 用户态微秒级自旋 (Microsecond Busy-Spin)
在 `nos_scheduler_run_loop` 中，线程在处理完一轮消息后，**不会立刻陷入 `epoll_wait` 阻塞**。
取而代之的是，线程会进入一段极短的 `__asm__ volatile("pause")` 循环自旋。
*   **命中率极高**：在高并发流水线中，下一次消息通常在几微秒内到达。由于线程在用户态自旋，它可以直接从内存队列 `msg_queue` 中取走新消息，完全避开 `epoll`。

### 2.2 状态感知与条件通知 (State-Aware Conditional Notify)
引入 `_Atomic int is_sleeping` 标志。
*   **发送方策略**：`nos_transport_local_send` 在把消息推入公共队列后，通过原子指令检查目标线程是否处于睡眠状态。
    *   `is_sleeping == 1`：触发 `write(eventfd)` 唤醒内核中的目标线程。
    *   `is_sleeping == 0`：跳过 `write` 系统调用。因为接收方处于活跃或自旋状态，它会在稍后自行从内存队列中摘取消息。

### 2.3 无锁通知句柄 (eventfd)
底层唤醒机制从基于字节流的 `pipe` 升级为专为事件通知设计的 `eventfd`，以最少的资源占用 (1 个 FD) 完成状态通知。

## 3. 副作用与局限性 (Trade-offs)

这种机制是用 **CPU 周期换取极致低时延**：
1.  **CPU 负载虚高**：在无负载时，线程也会短暂自旋，阻碍 CPU 核心进入低功耗状态，增加能耗和发热。
2.  **调度器竞争**：在共享或核心数受限（如 1 核或双核虚拟机）的普通 Linux 环境下，过长的忙等会抢占其他业务进程或同节点的低频 IO 处理任务（如定时器或跨进程 Socket），造成“饥饿”与响应抖动。
3.  **弱一致性竞态隐患**：在极高并发边缘场景下，`is_sleeping` 标志的状态转换存在极小概率的竞态，需要严格的内存屏障 (Memory Fence) 保驾护航。

## 4. 通用化与资源配置 (Generalization)

为适应从“资源受限物联网节点”到“极速数据中心网关”的不同场景，该优化必须是可配置的。

通过配置 `busy_poll_cycles` (自旋周期数)，可以平滑切换策略：

| 配置模式 | busy_poll_cycles | 场景与特性 |
| :--- | :--- | :--- |
| **标准阻塞模式** | `0` | **特性**：0 自旋，完全依赖 `epoll_wait`，CPU 利用率最低。<br>**适用**：边缘设备、低频管理节点、共享 CPU 资源的主机。 |
| **均衡模式 (默认)**| `100 - 500` | **特性**：提供几十纳秒的缓冲期，应对突发微量毛刺，平稳时立即休眠。<br>**适用**：一般性服务。 |
| **极速模式** | `5000 - 20000`| **特性**：微秒级自旋，实现高吞吐和 ns 级时延，CPU 会呈现高负载。<br>**适用**：物理绑核 (CPU Pinning) 的高性能业务、专有网络网关。 |
