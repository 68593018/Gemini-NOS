# 组件与服务模型设计 (草案)

## 1. 核心定义

### 1.1 组件 (Component)
*   **定义：** 系统的最小运行逻辑单元。
*   **特征：**
    *   **独立性：** 拥有私有的配置和状态数据。
    *   **受控性：** 其生命周期（加载、初始化、启动、停止）由所属线程的调度器管理。
    *   **非阻塞：** 必须在短时间内完成事件处理并返回，严禁执行阻塞操作。

### 1.2 服务 (Service)
在系统中，根据使用场景和性能要求的不同，服务被划分为两种类型：混合服务模型（Mixed Service Model）。

#### 1.2.1 嵌入式服务 (Embedded Service)
*   **隐喻：** 类似于 SQLite 等嵌入式数据库。
*   **特征：**
    *   **本地执行：** 以库（Library）的形式与调用者运行在同一进程和线程空间，直接进行函数调用。
    *   **同步极速：** 调用是同步的，几乎没有通信开销。
    *   **无感获取：** 不需要动态的服务发现。通常在系统初始化阶段，通过统一的接口列表（接口结构体）直接获取。
*   **适用场景：** 内存管理、基础算法库、纯计算逻辑、核心系统级轻量操作。

#### 1.2.2 远程服务 (Remote Service)
*   **隐喻：** 类似于 Redis 等独立的中心化服务进程。
*   **特征：**
    *   **异步隔离：** 运行在独立的组件、线程或进程中。强隔离，不会直接导致调用者崩溃。
    *   **消息驱动：** 必须通过异步消息总线（IPC/Message Broker）进行请求和响应交互，遵守非阻塞原则。
    *   **动态感知：** 调用者需要向总线订阅该服务的状态变化，只有服务处于 Online 状态时才能发起交互。
*   **适用场景：** 跨模块状态同步、设备驱动交互、路由下发等需要解耦和重试机制的复杂业务流程。

## 2. 调度模型 (Thread vs Component)
*   **多对一模型：** 一个系统线程可以承载多个组件。
*   **单点执行：** 任意时刻，一个线程内只有一个组件在运行，消除了线程内组件间的资源竞争。
*   **调度算法：** 基于事件驱动（如 Epoll 事件、定时器、内部消息队列优先级）。

## 3. 服务感知与交互
*   **服务注册：** 组件启动后向“服务管理器”注册其提供的 Service。
*   **服务发现：** 组件通过服务名称获取服务访问句柄或订阅服务状态。
*   **解耦原理：** 组件 A -> [感知] -> 服务 S <- [提供] <- 组件 B。A 与 B 之间无直接依赖。
