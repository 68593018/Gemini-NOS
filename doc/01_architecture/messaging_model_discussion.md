# NOS 消息模型与通信机制讨论记录

## 1. 服务与消息的核心定义
*   **一句话结论**：服务是由组件提供的逻辑功能实体，组件间通过约定格式的异步消息进行松耦合交互。
*   **交互规则**：
    | 特性 | 说明 |
    | :--- | :--- |
    | **消息信封** | 统一采用 `nos_service_msg_t` 作为头部，包含路由、类型及事务 ID。 |
    | **回复规则** | 由消息类型定义决定是“请求-响应”还是“单向推送”。 |
    | **事务关联** | 强制使用 `tx_id` (Transaction-ID) 匹配异步交互的请求与回复。 |

## 2. 消息类型语义 (Message Types)

| 类型 | 语义 | 适用场景 |
| :--- | :--- | :--- |
| **Unary (REQ/RSP)** | 1 请求 -> 1 响应。 | 关键配置下发、状态查询（需确保结果）。 |
| **Fire-and-Forget** | 只发不回，追求极致吞吐。 | 统计信息上报、非关键日志。 |
| **Subscription (PUB/SUB)** | 1 产生 -> N 接收。 | 接口状态变更通知、系统告警广播。 |

## 3. 深度讨论：高性能场景下的权衡 (以路由下发为例)

### 3.1 核心矛盾
*   **Unary 瓶颈**：在百万级路由更新时，等待每一个操作的响应（RSP）会严重拖慢系统收敛速度。
*   **Fire-and-Forget 风险**：虽然速度极快，但缺乏业务层确认，可能导致控制面与转发面的状态不一致。

### 3.2 潜在优化方向
1.  **异步流水线 (Pipelined Unary)**：支持在不等待前一个 RSP 的情况下连续发送多个 REQ，利用 `tx_id` 进行乱序匹配或滑动窗口确认。
2.  **分级回复**：采用 Fire-and-Forget 下发，但通过带外的事件通道（Event）汇报异常处理结果。
3.  **共享状态映射**：跳过消息封包，通过共享内存数据库（State DB）直接同步大规模路由表项。
