# NOS 依赖管理系统设计方案 (方案 1: 显式声明)

## 1. 核心理念
通过在配置阶段显式声明组件的“提供能力 (Provides)”与“资源需求 (Requires)”，实现系统拓扑的静态分析与动态编排。

## 2. 关键价值
*   **按需寻址**：进程仅加载其内部组件明确声明依赖的服务路由，降低内存占用，实现安全隔离。
*   **确定性启动**：基于拓扑排序 (Topological Sort) 自动推导组件间的启动顺序，消除初始化时的竞争与延迟。
*   **架构防腐**：在部署前静态拦截循环依赖及缺失服务，提高系统稳定性。

## 3. 配置规范 (YAML 示例)

```yaml
components:
  - name: "RouteComp"
    provides: [ SVC_ROUTING_V4 ] 
    requires: [ SVC_PORT_MGR ]
```

## 4. 实施逻辑流

### 4.1 离线阶段 (脚本层)
1.  **解析**：`gen_manifest.py` 扫描所有组件定义的 `provides` 和 `requires`。
2.  **验证**：
    *   检查 `requires` 的所有服务是否在系统内有对应的提供者。
    *   检测是否存在循环引用 (Circular Dependency)。
3.  **按需生成**：针对每个物理节点 (Proc)，仅导出该节点内组件所依赖的 `Remote Service` 路由信息。

### 4.2 运行阶段 (内核层)
1.  **上电序列**：`nos_node` 加载清单后，不再按数组索引启动，而是按照脚本计算好的 `Startup Level` 逐层激活组件。
2.  **状态监控**：当服务提供者异常离线时，基于依赖关系通知相关联的消费者进行容错处理（后续演进）。

---
*注：本设计旨在实现真正的“契约化部署”。*
