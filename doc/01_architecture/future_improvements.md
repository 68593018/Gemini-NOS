# NOS 平台架构后续改进建议 (草案)

## 1. 核心目标
通过增强数据一致性、通信性能和系统可观测性，将当前的 NOS 原型底座向生产级工业标准演进。

## 2. 改进项目矩阵

| 项目 | 核心价值 | 关键技术点 | 建议优先级 | 状态 |
| :--- | :--- | :--- | :--- | :--- |
| **KV 状态数据库** | 实现组件无状态化，支持故障恢复。 | 共享内存、Key-Value 模型、**订阅/发布机制**。 | **高** | **已实现基础框架与 Pub/Sub** |
| **线程安全日志** | 解决多线程日志交织，支持等级过滤。 | **非阻塞丢弃**、异步 Ring-Buffer、模块化标签。 | **高** | **已实现非阻塞与动态控制** |
| **共享内存 IPC** | 突破 UDS 拷贝瓶颈，实现零拷贝转发。 | POSIX Shm、无锁队列 (Lock-free Queue)。 | **中** | 待开发 |
| **事务级配置管理** | 确保系统配置的原子性与可回滚性。 | 二阶段提交、配置校验插件、差异化下发。 | **中** | 待开发 |

## 3. 详细说明

### 3.1 状态数据中心 (State Database)
*   **一句话结论**：建立“单一真理来源”，解耦业务逻辑与业务数据。
*   **简要说明**：目前各组件状态散落在私有堆内存中。引入基于内存的 KV Store 后，组件仅负责逻辑计算，关键状态同步至数据库，从而支持组件热升级后的无损状态恢复。

### 3.2 规范化日志系统 (Standardized Logging)
*   **一句话结论**：为多线程复杂系统的调试提供分类明确、等级可控、零业务干扰的观测视角。
*   **核心改进点**：
    1.  **异步化重构 (已完成基础框架)**：利用 Lock-free Ring-Buffer 将业务线程与 I/O 线程解耦。
    2.  **非阻塞丢弃策略 (Pending)**：在高负载导致 Ring-Buffer 溢出时，优先保护业务逻辑，采取“丢弃新日志并计数”的策略，严禁阻塞业务线程。
    3.  **延迟格式化 (Deferred Formatting)**：仅在后台线程进行字符串解析，进一步压低业务线程的 CPU 采样。
    4.  **运行时动态控制 (CLI Integration)**：将 `set_comp_level` 接口暴露给 CLI，支持 `log level <comp> <level>` 实时调测。
    5.  **文件滚动存储 (Persistence)**：支持日志落盘及基于大小/时间的自动翻转。

### 3.3 共享内存 IPC (Zero-copy IPC)
*   **一句话结论**：利用物理内存映射取代内核 Socket 拷贝，消除数据面吞吐瓶颈。
*   **简要说明**：在大流量转发场景下，UDS 带来的上下文切换和内存拷贝开销巨大。通过共享内存，进程间仅传递 Buffer 描述符，实现性能的质变。
