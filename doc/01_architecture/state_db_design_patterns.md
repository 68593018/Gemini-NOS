# NOS 状态存储架构设计探索 (State DB Design Patterns)

## 1. 架构演进背景
在探讨“KV 状态数据库”以实现组件无状态化时，我们最初提出了基于“变长/定长双引擎 Hash + 分段读写锁”的泛型 KV 存储方案。然而，站在**极致性能的工业级网络操作系统（如 Linux Kernel, DPDK, VPP, SONiC）**视角，仅仅使用“数据库的思维”是不够的，必须引入“数据面（Data Plane）的思维”。

本文档用于记录架构讨论中碰撞出的高级设计模式，为后续的正式开发提供理论依据。

## 2. 传统方案的局限性
泛型的 Hash KV DB（即使带有分段锁和内存池）在特定场景下存在局限：
*   **多核锁竞争**：在 64 核等高并发场景下，即使是读锁，Cache Line Bouncing 也会带来明显的性能衰减。
*   **网络匹配语义缺失**：路由表（如 IPv6 路由）的核心是**最长前缀匹配 (LPM)**，强制将其转化为固定长度或变长的 Hash Key 会极其低效且浪费内存。

## 3. 工业级进阶架构方案

### 3.1 机制升级：RCU (Read-Copy-Update) 无锁读取
*   **核心理念**：“读端绝对无锁，写端拷贝更新”。
*   **适用场景**：读写比例极度失衡（如 99% 读，1% 写）的高频访问表项（如 MAC 表）。
*   **机制描述**：写线程更新状态时不修改原节点，而是创建新节点并通过原子指针替换接入链表/树中。读线程无需获取任何锁即可访问，极大提升了多核并发读性能。

### 3.2 结构分离：引入网络专用数据结构
摒弃用一种泛型 KV 打天下的思路，针对不同的匹配语义提供专用的数据结构引擎：
*   **LPM 引擎 (Longest Prefix Match)**：专门针对路由表，底层实现为 **Radix Tree (基数树)** 或多级 **Trie 树 (如 DIR-24-8, DXR)**。
*   **精确匹配引擎 (Exact Match)**：专门针对五元组、MAC 地址等定长 Key，底层使用 **Cuckoo Hash** 或高度优化的流水线 Hash 表。

### 3.3 架构重构：控制面与数据面状态分离 (推荐路线)
将系统的状态存储需求彻底拆分为两个独立的子系统：

| 子系统 | NOS_CFG_DB (配置与控制面状态库) | NOS_FIB_ENG (转发信息库引擎) |
| :--- | :--- | :--- |
| **定位** | 面向控制面与管理面。 | 面向极致性能的数据面查表与转发。 |
| **存储特征** | 配置参数、组件运行上下文、统计信息。 | IPv4/v6 路由表、MAC 表、ACL 表。 |
| **底层结构** | 双引擎 Hash KV (变长字符串 + 定长结构)。<br>分段读写锁 (Segmented RW-Lock)。 | 专用表结构 (Radix Tree, Cuckoo Hash)。<br>无锁读取 (RCU 机制)。 |
| **API 形态** | 标准的 KV 接口 (`put`, `get`, `del`) | 领域专用的表 API (`add_route`, `lookup_mac`) |

## 4. 结论与后续规划
在原型开发阶段，我们可以优先实现 **NOS_CFG_DB (控制面状态库)**，它足以解决当前多个组件之间的状态共享和无状态化改造需求。

对于数据面相关的高频表项，我们将把上述“高级专用引擎”的设计思路作为技术储备，待系统开始承载真实的网络转发业务时，再进行独立的 `NOS_FIB_ENG` 设计与开发。
