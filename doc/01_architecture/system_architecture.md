# NOS 平台软件宏观架构设计 (草案)

## 1. 总体设计原则
*   **分层解耦：** 软硬件分离，业务逻辑与系统底座分离。
*   **数据驱动：** 状态数据与业务逻辑分离，通过集中式管理或标准化接口进行通信。
*   **模块化隔离：** 各功能域相对独立，通过标准 IPC 通信，降低故障扩散风险。

## 2. 系统架构图 (分层视图)

| 层级 | 核心职责 | 关键组件示例 |
| :--- | :--- | :--- |
| **管控应用层 (App/Mgmt)** | 对外接口、用户交互、高级协议。 | CLI, SNMP, BGP, OSPF, WebUI. |
| **网络服务层 (Network Services)** | 业务抽象与策略控制。 | VLAN Manager, Route Manager, ACL/QoS Policy. |
| **核心系统层 (Core System)** | 系统运行底座与基础组件。 | IPC Bus, Key-Value Store (Config/State), Log, HA. |
| **硬件抽象层 (HAL/SAI)** | 硬件适配与指令翻译。 | Switch SDK Wrapper, SAI Interface, Peripheral Driver. |

## 3. 核心功能域划分

| 功能域 | 重点验证方向 | 备注 |
| :--- | :--- | :--- |
| **基础设施 (Infra)** | 高性能消息总线、数据库同步机制。 | 项目启动的基石。 |
| **设备管理 (Device)** | 端口建模、热插拔处理、硬件监控。 | 涉及硬件抽象层的定义。 |
| **转发平面 (Forwarding)** | 路由表项(FIB)下发流程、L2/L3 同步机制。 | 验证数据面与控制面的协同。 |

## 4. 演进路线
1. **基础设施原型化：** 确定 IPC 和基本的数据模型。
2. **HAL 定义：** 抽象出基础的接口定义。
3. **协议/业务叠加：** 在稳固的底座上实现具体业务逻辑。
